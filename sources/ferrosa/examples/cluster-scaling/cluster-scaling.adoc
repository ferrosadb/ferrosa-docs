:toclevels: 3
:stylesheet: ../theme/ferrosa.css
:linkcss:

Ferrosa is designed to grow from a single development node toward clustered deployments without changing APIs.
Most databases force you to choose your deployment topology up front -- a single-node install that you later throw away and replace with a cluster, or a heavyweight multi-node setup that's overkill for development.
Ferrosa aims to reduce that tradeoff: start with one node on your laptop, evaluate pair mode for standby behavior, then test Raft-backed clustering as your operational requirements grow.

This tutorial walks through three stages of that journey: *standalone* (development), *pair mode* (standby evaluation), and *Raft cluster* (multi-node consensus evaluation).
At each transition, the demo exercises writes and schema changes so you can observe behavior during topology changes.
The data you inserted on a single node is still there after you've scaled to three -- no migrations, no re-bootstrapping, no downtime windows.

== Prerequisites

* *Docker and Docker Compose* -- see the link:cluster-setup.html[Cluster Setup Guide] if you need to install Docker
* *cqlsh* -- `pip install cqlsh`
* About *4 GB of RAM* available for the three nodes plus the object store

== Act 2: Pair Mode -- Low-Volume Production

Pair mode adds a hot standby that receives replicated writes in real time.
If the primary fails, the secondary can serve reads immediately.
Think of it like PostgreSQL streaming replication, but it also replicates DDL -- schema changes propagate automatically.

=== Add node2

Scale from standalone to pair mode by layering the pair override file:

[source,bash]
----
docker compose -f docker-compose.yml -f docker-compose.pair.yml up -d
----

This does two things: it restarts node1 with `FERROSA_CLUSTER_MODE=pair`, and it launches node2 as the secondary.
Node2 connects to node1, receives the current schema and data, and begins replicating writes.

IMPORTANT: Writes continue uninterrupted during this transition. Node1 accepts CQL commands throughout the mode change. If you have a background writer running (like `run-demo.sh` provides), it will not see any errors.

Watch the pair form:

[source,bash]
----
docker compose logs -f node1 node2
----

You should see node2 complete its handshake and report "Pair mode established with node1".
Press `Ctrl+C` to stop following logs.

You should see the new `user_id` column and `events_by_user` index on both nodes.

NOTE: In traditional Cassandra, schema changes propagate via gossip and can temporarily disagree across nodes. In Ferrosa pair mode, DDL is replicated synchronously -- the secondary's schema is always consistent with the primary's.

[[act-3]]
== Act 3: Raft Cluster -- Full Production

When your traffic outgrows a two-node pair, add a third node to form a Raft consensus cluster.
Three nodes give you:

* *Fault tolerance* -- the cluster continues operating if any single node fails
* *Tunable consistency* -- choose between `ONE`, `QUORUM`, or `ALL` per query
* *Consensus-backed metadata* -- schema changes and topology updates go through Raft, guaranteeing agreement

=== Add node3

Layer all three compose files to bring up the full cluster:

[source,bash]
----
docker compose -f docker-compose.yml -f docker-compose.pair.yml -f docker-compose.cluster.yml up -d
----

This transitions all three nodes to `FERROSA_CLUSTER_MODE=cluster`.
Node3 joins, a Raft leader election completes, and the cluster stabilizes.

[source,bash]
----
docker compose logs -f node1 node2 node3
----

Watch for these log messages:

. Node3 connects to node1 (the seed)
. Raft leader election completes -- one of the three nodes becomes leader
. All three nodes report "Cluster mode active, 3 nodes in ring"

----

Connect to each node and check:

* *Row counts* -- the `app.events` table should contain all rows inserted across all three phases
* *Schema agreement* -- `DESCRIBE TABLE app.events` should show identical output on all three nodes, including the columns and indexes added during pair mode and cluster mode
* *Cluster health* -- query the web API on any node:

[source,bash]
----
curl -s http://localhost:9090/api/cluster/status | python3 -m json.tool
----

All three nodes should show `"status": "UP"`.

== Running the Demo

The `run-demo.sh` script automates the sequence above -- starting each phase, running CQL scripts, maintaining a background writer during transitions, and asserting correctness at each stage.

[source,bash]
----
bash run-demo.sh
----

The script prints progress for each act and exits with a non-zero status if any assertion fails.

== Teardown

Remove all containers and volumes:

[source,bash]
----
docker compose -f docker-compose.yml -f docker-compose.pair.yml -f docker-compose.cluster.yml down -v
----

This deletes all data. The S3 bucket, node volumes, and container state are all removed.
